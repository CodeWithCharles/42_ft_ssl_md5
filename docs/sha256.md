# SHA-256 (Secure Hash Algorithm, 256 bits)

> Prérequis : lire **[Merkle–Damgård](merkle-damgard.md)** d'abord. On ne décrit
> ici que ce qui est **spécifique à SHA-256**. La grande différence avec MD5 :
> **big-endian**, **8 mots d'état**, et un **message schedule** de 64 mots.

## Fiche technique

| Propriété              | Valeur                                   |
|------------------------|------------------------------------------|
| Taille du digest       | 256 bits = **32 octets**                 |
| Mots d'état            | **8** mots de 32 bits (a…h)              |
| Taille de bloc         | 512 bits = **64 octets**                 |
| Champ longueur         | 64 bits, **big-endian**                  |
| Endianness             | **big-endian** partout                   |
| Nombre de rondes       | **64**                                   |

## État initial (IV)

Les 32 bits de tête des parties fractionnaires des **racines carrées** des 8
premiers nombres premiers (2, 3, 5, 7, 11, 13, 17, 19) :

```
   h0 = 0x6a09e667   h1 = 0xbb67ae85   h2 = 0x3c6ef372   h3 = 0xa54ff53a
   h4 = 0x510e527f   h5 = 0x9b05688c   h6 = 0x1f83d9ab   h7 = 0x5be0cd19
```

→ Code : `g_sha256_iv`.

## Les constantes K

**K[0..63]** : les 32 bits de tête des parties fractionnaires des **racines
cubiques** des 64 premiers nombres premiers. → `g_sha256_k`.

## Les fonctions

SHA-256 utilise `ROTR(x, n)` = rotation **droite** (pas gauche comme MD5) et
`SHR(x, n)` = décalage droit (avec des zéros). Six fonctions :

```
   Ch(e,f,g) = (e ∧ f) ⊕ (¬e ∧ g)                        « choisit » f ou g selon e
   Maj(a,b,c) = (a ∧ b) ⊕ (a ∧ c) ⊕ (b ∧ c)              bit majoritaire
   Σ0(a) = ROTR(a,2)  ⊕ ROTR(a,13) ⊕ ROTR(a,22)          (BSIG0)
   Σ1(e) = ROTR(e,6)  ⊕ ROTR(e,11) ⊕ ROTR(e,25)          (BSIG1)
   σ0(x) = ROTR(x,7)  ⊕ ROTR(x,18) ⊕ SHR(x,3)            (SSIG0)
   σ1(x) = ROTR(x,17) ⊕ ROTR(x,19) ⊕ SHR(x,10)           (SSIG1)
```

- Les **Σ** (majuscules) servent dans la compression.
- Les **σ** (minuscules) servent à étendre le message (schedule).

## Le message schedule W[0..63]

Contrairement à MD5 qui réutilise les 16 mots du bloc dans un ordre permuté,
SHA-256 **dérive 64 mots** à partir des 16 mots du bloc :

```
   W[t] = mot big-endian n°t du bloc                       pour t = 0..15
   W[t] = σ1(W[t-2]) + W[t-7] + σ0(W[t-15]) + W[t-16]      pour t = 16..63
```

Ça « étale » les 512 bits d'entrée sur les 64 rondes, chaque ronde consommant un
`W[t]` différent. → Code : `sha256_schedule`.

## La compression d'un bloc

On copie l'état dans 8 registres `a, b, c, d, e, f, g, h`, puis **64 rondes** :

```
   T1 = h + Σ1(e) + Ch(e,f,g) + K[t] + W[t]
   T2 = Σ0(a) + Maj(a,b,c)
   h = g
   g = f
   f = e
   e = d + T1
   d = c
   c = b
   b = a
   a = T1 + T2
```

Deux « quantités » `T1` (mélange la moitié droite de l'état + la clé de ronde + le
message) et `T2` (mélange la moitié gauche), puis **décalage en cascade** des
8 registres. Additions modulo 2³².

> Dans le code, `T2` est fusionné dans le calcul final : après les décalages,
> `v[1]=a`, `v[2]=b`, `v[3]=c` (valeurs d'avant rotation), donc
> `v[0] = T1 + Σ0(a) + Maj(a,b,c)`. Ça évite une variable temporaire.

### Feed-forward

Comme MD5, après les 64 rondes on rajoute les registres à l'état :

```
   h0 += a ; h1 += b ; … ; h7 += h
```

→ Code : `sha256_transform`.

## Sérialisation → digest

Les 8 mots finaux `h0..h7` écrits **en big-endian**, bout à bout :

```
   digest = BE(h0) ‖ BE(h1) ‖ … ‖ BE(h7)     (32 octets)
```

→ Code : `md_serialize32(ctx, digest, SHA256_WORDS, MD_BE)`.

## Exemple déroulé : `SHA256("abc")`

- Message : `61 62 63` (24 bits).
- Padding :
  ```
  61 62 63 80 00 00 … 00   (jusqu'à l'octet 55)
  00 00 00 00 00 00 00 18  (octets 56–63 : longueur = 24 bits, en BIG-endian)
  ```
  Compare avec MD5 : ici le `18` est en **dernier** octet (big-endian).
- Résultat :
  ```
  ba7816bf8f01cfea414140de5dae2223b00361a396177a9cb410ff61f20015ad
  ```

## MD5 vs SHA-256 en un tableau

| Aspect                | MD5                    | SHA-256                     |
|-----------------------|------------------------|-----------------------------|
| Digest                | 16 octets              | 32 octets                   |
| Mots d'état           | 4                      | 8                           |
| Endianness            | little-endian          | big-endian                  |
| Rotation              | gauche (`rotl`)        | droite (`ROTR`)             |
| Message               | 16 mots permutés       | schedule de 64 mots (σ0/σ1) |
| Ronde                 | 1 fonction (F/G/H/I)   | T1/T2 (Σ0/Σ1/Ch/Maj)        |
| Sécurité              | **cassé** (collisions) | sûr aujourd'hui             |

Dans le code, **tout le reste est partagé** (`md_core`) : padding, absorption,
sérialisation, streaming. Ajouter SHA-256 après MD5 n'a coûté qu'un fichier
`sha256.c` + une ligne dans la table de dispatch — c'est le test de réussite de
l'architecture.

## Bonus : SHA-224

SHA-224 est **SHA-256 avec deux changements seulement** :

1. un **IV différent** (8 autres constantes de 32 bits) ;
2. le digest **tronqué aux 7 premiers mots d'état** (h0…h6), soit 224 bits =
   28 octets ; on jette h7.

Tout le reste — message schedule, compression, constantes K, big-endian,
padding — est **identique** à SHA-256. Dans le code, SHA-224 vit dans le même
`sha256.c` et **réutilise `sha256_transform` / `sha256_update`** ; seuls
`sha224_init` (l'IV) et `sha224_final` (troncature à `SHA224_WORDS = 7`) lui
sont propres. Même moteur, juste re-paramétré — une nouvelle preuve que l'archi
tient (ni `md_core`, ni `exec`, ni `output` n'ont bougé).

IV de SHA-224 (32 bits de tête des racines carrées des 9ᵉ→16ᵉ premiers) :

```
   h0 = c1059ed8   h1 = 367cd507   h2 = 3070dd17   h3 = f70e5939
   h4 = ffc00b31   h5 = 68581511   h6 = 64f98fa7   h7 = befa4fa4
```

## Test rapide

```sh
./ft_ssl sha256 -s "abc"        # SHA256 ("abc") = ba7816bf…f20015ad
printf "" | ./ft_ssl sha256     # (stdin)= e3b0c442…7852b855
./ft_ssl sha224 -s "abc"        # SHA224 ("abc") = 23097d22…e36c9da7
printf "" | ./ft_ssl sha224     # (stdin)= d14a028c…c5b3e42f
```
