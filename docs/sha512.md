# SHA-512 & SHA-384 (bonus, moteur 64 bits)

> Prérequis : lire **[SHA-256](sha256.md)** d'abord. SHA-512 en est la **variante
> 64 bits** : exactement la même structure (schedule, T1/T2, feed-forward), mais
> mots de 64 bits, bloc deux fois plus grand, et quelques constantes de rotation
> différentes.

## Fiche technique

| Propriété        | SHA-256            | SHA-512                    |
|------------------|--------------------|----------------------------|
| Taille du digest | 32 octets          | **64 octets**              |
| Mots d'état      | 8 × **32** bits    | 8 × **64** bits            |
| Taille de bloc   | 64 octets          | **128 octets**             |
| Champ longueur   | 64 bits (8 o)      | **128 bits (16 o)**        |
| Nombre de rondes | 64                 | **80**                     |
| Endianness       | big-endian         | big-endian                 |

## Ce qui change par rapport à SHA-256

1. **Mots de 64 bits** partout : IV, état, schedule, K sont des `uint64_t`.
   `ROTR` tourne sur `64 - n` (le piège n°1 quand on copie depuis SHA-256).
2. **Rotations différentes** dans les fonctions Σ/σ :
   ```
   Σ0(x) = ROTR(x,28) ^ ROTR(x,34) ^ ROTR(x,39)
   Σ1(x) = ROTR(x,14) ^ ROTR(x,18) ^ ROTR(x,41)
   σ0(x) = ROTR(x, 1) ^ ROTR(x, 8) ^ SHR(x, 7)
   σ1(x) = ROTR(x,19) ^ ROTR(x,61) ^ SHR(x, 6)
   ```
3. **80 rondes** (au lieu de 64), **80 constantes K** de 64 bits, message
   schedule W de **80 mots**.
4. **Bloc de 128 octets**, **champ longueur de 128 bits**.

`Ch`, `Maj`, la mécanique T1/T2 et le feed-forward sont **identiques** à SHA-256
(juste en 64 bits). IV de SHA-512 = 64 bits de tête des racines carrées des
8 premiers premiers :

```
   h0 = 6a09e667f3bcc908   h1 = bb67ae8584caa73b
   h2 = 3c6ef372fe94f82b   h3 = a54ff53a5f1d36f1
   h4 = 510e527fade682d1   h5 = 9b05688c2b3e6c1f
   h6 = 1f83d9abfb41bd6b   h7 = 5be0cd19137e2179
```

## Le piège du padding 128 bits (bug réel corrigé ici)

Le champ longueur de SHA-512 fait **128 bits**, mais un message réaliste tient sur
64 bits → les **8 octets de poids fort du champ doivent être à zéro**.

Notre `md_finalize` remplissait les zéros jusqu'à `block_size - len_bytes` (= 112)
et `write_length` n'écrivait que les **8 octets de poids faible** (`[120..127]`).
Résultat : les octets **`[112..119]`** n'étaient ni zérotés ni écrits → **garbage
de pile** → hash faux (et lecture non initialisée).

- **SHA-256** : `len_bytes = 8`, le remplissage s'arrêtait pile où `write_length`
  reprenait → pas de trou, jamais vu le bug.
- **SHA-512** : `len_bytes = 16` → trou de 8 octets.

**Correctif** (`md_core.c`) : zéro-remplir tout le reste du bloc **jusqu'à
`block_size`**, puis laisser `write_length` réécrire les 8 octets de poids faible
de la longueur par-dessus. Robuste pour n'importe quel `len_bytes`. Les frontières
de padding **111 / 112** (SHA-512) sont désormais des cas de test dédiés
(`make test`).

## SHA-384 : SHA-512 tronqué

Comme SHA-224 l'est à SHA-256 : **même moteur**, deux changements seulement —
un **IV différent** et le digest **tronqué aux 6 premiers mots** (h0…h5), soit
384 bits = 48 octets. Dans le code, SHA-384 vit dans `sha512.c` et **réutilise
`sha512_transform` / `sha512_update`** ; seuls `sha384_init` (IV) et
`sha384_final` (`md_serialize64` sur `SHA384_WORDS = 6`) lui sont propres.

IV de SHA-384 :

```
   h0 = cbbb9d5dc1059ed8   h1 = 629a292a367cd507
   h2 = 9159015a3070dd17   h3 = 152fecd8f70e5939
   h4 = 67332667ffc00b31   h5 = 8eb44a8768581511
   h6 = db0c2e0d64f98fa7   h7 = 47b5481dbefa4fa4
```

## Côté architecture

Le seul ajout à `md_core` pour toute la famille 64 bits = **`md_serialize64`**
(miroir de `md_serialize32`, mots de 8 octets). `md_absorb` et `md_finalize`
étaient déjà génériques (`block_size` / `len_bytes` en paramètres, `MD_LEN128`
prévu de longue date). SHA-512 + SHA-384 = **1 fichier `sha512.c` + 1 fonction de
sérialisation + 2 lignes de dispatch**.

## Test rapide

```sh
./ft_ssl sha512 -s "abc"   # SHA512 ("abc") = ddaf35a1…fa54ca49f
printf "" | ./ft_ssl sha512 # (stdin)= cf83e135…927da3e
./ft_ssl sha384 -s "abc"   # SHA384 ("abc") = cb00753f…34c825a7
```
